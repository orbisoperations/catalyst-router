Architecture Evolution

Catalyst: From Services to Environment

Catalyst is evolving from an SDK-driven service mesh into a transparent network environment — a QUIC-backed overlay you route through, not code against.

Clients use capnweb RPC to register routes & allocate ports
Control Plane

Envoy (xDS) Port Allocator capnweb RPC
Integration model: Applications import an SDK, open a capnweb WebSocket session, call getDataChannelClient(), and register routes with allocated ports. Tight coupling.
Next
3 Planes · Network Model
Applications use DNS + standard networking — no SDK required
Management Plane
Orchestrator PKI Token Cedar

Integration model: Applications resolve FQDNs via DNS, traffic routes through Envoy using SNI / domain filtering. Catalyst is the network — like a web proxy or NAT gateway.

📡 Appliance Mode

Deployed via K8s onto hardware or virtual appliance. Acts as a web proxy / NAT gateway on the network — configured as the default route for a site. Services behind it reach peered destinations transparently. Think squid proxy, but for your entire Catalyst mesh.

☸️ K8s Sidecar Mode

Deployed alongside workloads in an existing cluster. Hooks into the K8s ecosystem — takes over CoreDNS for mesh-aware resolution, integrates with service mesh primitives. No separate appliance needed; Catalyst becomes part of the cluster's network fabric.

Addendum — Work To Be Done

Implementation Roadmap

Adoption work required to move from the current codebase to the 3-plane network architecture. Each item identifies the component, what changes, and how it differs between the k3s appliance and k8s cloud deployment.

NEW ADOPTION Smallstep step-ca — Private ACME CA

Replace the existing custom PKI implementation with step-ca as the private Certificate Authority. step-ca provides an ACME HTTP-01 provisioner so services can request certificates using standard ACME clients — no custom cert-issuance code needed. The orchestrator's PIK endpoint proxies ACME challenges through the data plane. step-ca handles ACME HTTP-01 challenges directly in the management plane. The orchestrator provisions DNS records first (data-channel-first flow), enabling the HTTP-01 challenge to resolve.

📡 k3s / Standalone Appliance
--kms yubikey for hardware key storage (YubiKey PIV via step-kms-plugin) or --kms softkms for local filesystem keys. Single replica Deployment in catalyst-system. Root CA key never leaves the hardware token.
☸️ k8s / Cloud Environment
--kms cloudkms (GCP) or --kms awskms (AWS) for cloud-managed key storage. Root CA key managed by provider HSM. Can run HA with multiple replicas if backed by shared storage.
Work involved: Deploy step-ca container with ACME provisioner config. Write provisioner templates for service certs (short-lived, auto-renew) and client certs (CLI-issued, certificate-bound tokens). Wire PIK to proxy HTTP-01 challenges. Orchestrator provisions DNS + xDS on data channel registration; step-ca validates HTTP-01 against the live DNS record. Token validation is Envoy-native via Cedar ext_authz — no separate Token API service. Remove existing custom CA code from auth service.
NEW ADOPTION CoreDNS — Mesh DNS Resolution

Deploy CoreDNS as the authoritative DNS server for mesh FQDNs. The orchestrator writes zone files as peers announce routes via BGP-like UPDATE messages — CoreDNS watches those files and serves the records. Non-mesh queries forward upstream. This replaces any custom DNS resolution and gives every application standard DNS-based service discovery.

📡 k3s / Standalone Appliance
file / auto plugin — orchestrator writes standard zone files to a shared volume (/zones). CoreDNS watches via inotify. No etcd dependency (k3s uses SQLite). Standalone deployment in catalyst-data.
☸️ k8s / Cloud Environment
Two options: (a) Add a custom server block to the existing cluster CoreDNS — mesh zone delegates to Catalyst CoreDNS, everything else stays default. (b) Use the etcd plugin — orchestrator writes records directly to cluster etcd instead of zone files. Option (b) avoids a second CoreDNS but couples to etcd availability.
Work involved: Add zone-file writer to orchestrator (triggered on BGP route table changes). Define mesh zone format (e.g., *.mesh.catalyst.local). Deploy CoreDNS with Corefile configuring auto + forward plugins. For k8s: write etcd adapter as alternative to file plugin. Configure DHCP/NAT to hand out CoreDNS as the resolver for appliance-mode clients.
REFACTOR Envoy — Filesystem xDS (replace gRPC ADS)

Replace the custom gRPC xDS control plane (envoy-service) with filesystem-based xDS. The orchestrator writes xDS JSON files (LDS, CDS, SDS) to a shared volume — Envoy watches via path_config_source + inotify and hot-reloads. This eliminates a standalone service, a gRPC server, and all connection lifecycle management. Same pattern as the CoreDNS zone file approach — the orchestrator is the single writer, everything else watches.

📡 k3s / Standalone Appliance
Shared emptyDir volume between orchestrator and Envoy DaemonSet. Orchestrator writes lds.json, cds.json, sds.json on every route table change. Single node — volume locality is trivial. Envoy bootstrap uses path_config_source with watched_directory.
☸️ k8s / Cloud Environment
Same filesystem approach. For multi-node clusters, orchestrator runs as a DaemonSet (one per node) and writes xDS locally. Alternatively, a central orchestrator can write to a PersistentVolume with ReadWriteMany (NFS/EFS) or push via a lightweight config-sync sidecar. The volume strategy depends on cluster topology — single-writer-per-node is simplest.
Work involved: Refactor orchestrator's xDS generation to write JSON files instead of building in-memory snapshots. Remove apps/envoy/src/xds/control-plane.ts (gRPC ADS server) and apps/envoy/src/xds/resources.ts (snapshot builder). Write new xds-writer module in orchestrator. Create Envoy bootstrap config template with path_config_source. Add health check: orchestrator watches Envoy admin /config_dump to detect rejected configs (replaces gRPC ACK/NACK).
REFACTOR Envoy — SNI Filter Chains (replace port allocation)

Replace the current port-per-channel model with a single listener on :443 using SNI-based filter_chain_match.server_names for per-service routing. Each service gets a unique FQDN; Envoy matches the TLS SNI to the right filter chain, terminates TLS, applies Cedar ext_authz, and forwards to the local upstream. No more port allocator, no more port pool exhaustion.

📡 k3s / Standalone Appliance
Single Envoy listener on :443. DHCP/NAT directs all site HTTPS traffic here. Each registered service gets a filter chain matched by SNI. Wildcard chain for QUIC peer-to-peer upstream on :8443.
☸️ k8s / Cloud Environment
Same SNI model. Envoy sits behind a cloud LoadBalancer (NLB/GLB) or k8s Ingress. SNI passthrough at the LB layer lets Envoy handle TLS termination and Cedar policy. Integrates with existing cluster ingress without conflicting.
Work involved: Remove apps/envoy/src/port-allocator.ts and port pool logic from orchestrator bus. Update xDS writer to generate LDS with SNI filter chain match entries instead of per-port listeners. Update route propagation BGP messages to carry FQDNs (already partially implemented). Wire SDS entries for per-service TLS contexts (cert + key from step-ca).
INTEGRATION Cedar ext_authz — Policy at Every Hop

Wire Cedar as an Envoy ext_authz gRPC filter on every filter chain. Every TLS termination point — local Envoy and remote Envoy — independently evaluates Cedar policy before forwarding traffic. No trust delegation, no passthrough. Cedar policies are synced from the orchestrator via BGP route attributes.

📡 k3s / Standalone Appliance
Cedar engine runs as a single-replica Deployment in catalyst-system. Envoy's ext_authz filter points to Cedar via cluster-internal DNS. Low latency — same node, same network namespace.
☸️ k8s / Cloud Environment
Same architecture. Cedar can scale horizontally if policy evaluation becomes a bottleneck (stateless evaluator, policies loaded from orchestrator). Consider sidecar deployment alongside Envoy for latency-sensitive paths.
Work involved: Existing Cedar implementation (ADR 0015) needs a gRPC ext_authz adapter. Define policy schema for SNI-based routing decisions (principal = client cert identity, resource = target FQDN, action = protocol/method). Add policy distribution to BGP UPDATE messages. Add ext_authz filter to every generated LDS filter chain in the xDS writer.
INTEGRATION Envoy QUIC Cluster — Peer-to-Peer Transit

Catalyst peers form an Envoy cluster connected via QUIC transport. Traffic between peers flows as QUIC streams within the cluster — not individual tunnels. Each peer's Envoy terminates TLS independently and applies Cedar policy before forwarding to local services. The orchestrator's BGP-like route exchange drives cluster membership.

📡 k3s / Standalone Appliance
QUIC upstream on :8443. Peers discovered via orchestrator route exchange. CDS entries generated per-peer with QUIC transport socket. mTLS between peers using step-ca issued node certificates.
☸️ k8s / Cloud Environment
Same QUIC mesh. Cloud LB must allow UDP for QUIC (NLB on AWS, external passthrough on GCP). For cross-region peers, QUIC's built-in connection migration handles path changes gracefully.
Work involved: Configure Envoy QUIC upstream transport socket in CDS entries. Generate per-peer cluster definitions in xDS writer (triggered by BGP peer state changes). Test QUIC connection establishment with mTLS. Verify Cedar policy enforcement on both sides of the QUIC hop. Benchmark QUIC vs TCP for Catalyst traffic patterns.
REFACTOR Orchestrator — Single Writer Pattern

The orchestrator becomes the single source of truth that drives both DNS and Envoy configuration. On every route table change (peer join, service registration, route withdrawal), it atomically writes two sets of files to shared volumes: zone files for CoreDNS and xDS JSON for Envoy. Both consumers watch via inotify. This pattern eliminates the need for separate xDS and DNS translation services.

Applies to both environments
Route table change → orchestrator writes /zones/*.db (CoreDNS) + /xds/{lds,cds,sds}.json (Envoy) → consumers hot-reload via inotify. Atomic writes via rename to prevent partial reads. On k3s, volumes are emptyDir. On k8s, volume strategy depends on topology (emptyDir for per-node, PVC/NFS for centralized).
Work involved: Refactor orchestrator bus (apps/orchestrator/src/v2/bus.ts) to emit file writes instead of in-process Envoy sync calls. Implement atomic write-then-rename for both zone files and xDS JSON. Add file-write health metrics. Remove direct Envoy gRPC sync from orchestrator. Integration test: route change → file written → Envoy/CoreDNS pick up within <1s.
Addendum — Protocol Flows

How It All Comes Together

Three sequences showing the lifecycle: system bootstrap, first service onboarding, and first client connection. Each step identifies the component responsible and what must be in place.

Flow A System Bootstrap — Cold Start to Ready

A fresh Catalyst node boots on K8s. This sequence shows the dependency chain: what starts first, what waits for what, and when the system is ready to accept services and clients. The orchestrator is the last to become ready because it depends on everything else.
Phase 1 — Infrastructure & PKI Bootstrap
A1 k3s / k8s Kubernetes runtime starts
k3s server initializes (SQLite datastore on appliance) or k8s node joins existing cluster. Kubelet begins scheduling pods. Network CNI ready.
A2 step-ca Private CA initializes
step-ca pod starts in catalyst-system. Loads root CA key from KMS backend (yubikey on appliance, cloudkms on cloud). Initializes ACME HTTP-01 provisioner. Exposes :9000 for ACME + API. Everything else depends on this — no certs, no TLS, no mesh.
A3 step-ca Node identity cert issued
Orchestrator requests a node identity certificate from step-ca via ACME. This cert identifies the node in peer-to-peer mTLS and signs BGP route messages. FQDN: node-{id}.mesh.catalyst.local.
A4 token-service Token service starts
Obtains its own service cert from step-ca. Generates ES384 signing key (or loads from persistent storage). Exposes JWKS at /.well-known/jwks.json. Ready to mint JWTs with cnf.x5t#S256 certificate binding.
A5 cedar-engine Cedar policy engine starts
Loads default policy set (deny-all baseline + system policies). Exposes gRPC ext_authz on :3002. Waits for orchestrator to push service-specific policies via BGP route attributes.
Phase 2 — Data Plane Boot
A6 CoreDNS DNS server starts (empty zone)
CoreDNS pod starts in catalyst-data. Loads Corefile with auto plugin watching /zones directory. Zone dir is empty — no mesh records yet. forward plugin proxies non-mesh queries upstream. Listening on 53/udp, 53/tcp.
A7 Envoy Envoy starts (minimal bootstrap)
Envoy DaemonSet pod starts in catalyst-data. Loads bootstrap config with path_config_source watching /xds directory. xDS files don't exist yet — Envoy runs with static admin listener only. ext_authz filter points to Cedar engine. Listening on :443 (no filter chains yet) and :8443 (QUIC peer, no clusters yet).
Phase 3 — Orchestrator Brings It All Together
A8 orchestrator Orchestrator starts — writes initial config
Orchestrator pod starts in catalyst-system. Verifies step-ca reachable, token-service healthy, Cedar responding. Initializes empty route table (RIB). Writes initial zone file to /zones/mesh.catalyst.local.db (SOA + NS only). Writes initial xDS files to /xds/{lds,cds,sds}.json (base listener on :443, no filter chains). CoreDNS and Envoy hot-reload via inotify.
A9 orchestrator Peer discovery & BGP session establishment
Orchestrator announces itself to configured peers via BGP OPEN using node identity cert (mTLS). If peers exist: exchanges KEEPALIVE, receives UPDATE messages with remote FQDNs, nodePaths, CA thumbprints. On first boot with no peers: skips — node is the first in the mesh.
A10 orchestrator Route table → DNS zones + xDS files (peer routes)
For each peer route received: adds A/AAAA record to zone file (FQDN → local Envoy), adds CDS entry for peer Envoy cluster (QUIC upstream), adds SDS entry with peer CA trust bundle. Writes atomically (rename). Envoy now has QUIC clusters to peers. CoreDNS now resolves remote FQDNs. System is ready for services.
A11 DHCP/NAT appliance only NAT gateway activates
DHCP/NAT DaemonSet configures iptables: mesh FQDNs → Envoy :443, DNS → CoreDNS :53, all other traffic → upstream internet. Hands out CoreDNS as resolver to DHCP clients. Site network is now routed through Catalyst.

Flow B First Service Onboarding — Webapp Gets a Certificate Data Channel Registration

A webapp deployed on the node obtains a TLS certificate via ACME HTTP-01, registers with the mesh, and becomes reachable from all peers via DNS + Envoy SNI routing. No SDK, no custom code — standard ACME client (certbot, lego, acme.sh, etc.). A service is registered as a data channel via the orchestrator API. The orchestrator provisions DNS, xDS, and a TLS certificate — making the service routable before it ever touches ACME. Optionally, the service can then request its own ACME cert using the already-live DNS record.
Phase 1 — ACME Certificate Issuance
B1 webapp Webapp starts & initiates ACME
Webapp runs in catalyst-apps (or any namespace). Uses a standard ACME client library to contact step-ca's ACME directory at https://step-ca.catalyst-system:9000/acme/acme/directory. Requests a cert for its desired FQDN: myapp.node-a.mesh.catalyst.local.
B2 step-ca ACME order created — HTTP-01 challenge issued
step-ca creates a new ACME order and returns an HTTP-01 challenge. Provides a token value. Webapp must prove domain control by serving keyAuthorization at http://myapp:port/.well-known/acme-challenge/<token>.
B3 webapp Webapp serves challenge token
ACME client starts a temporary HTTP listener (or hooks into existing server) to serve the keyAuthorization value at the well-known path. This is standard ACME — no Catalyst-specific code.
B4 step-ca → webapp PIK validates HTTP-01 challenge
step-ca (via its HTTP-01 validation) makes a GET request to the webapp at /.well-known/acme-challenge/<token>. This goes over the cluster network (k8s service DNS). Webapp returns keyAuthorization — challenge passes.
B5 step-ca → webapp Certificate issued
step-ca signs the certificate for myapp.node-a.mesh.catalyst.local. Webapp downloads cert + chain via ACME finalize. Webapp now has a valid TLS cert signed by the mesh CA. Short-lived (e.g. 24h), auto-renews via ACME.
Phase 2 — Mesh Registration
B6 orchestrator Orchestrator detects new certificate
Orchestrator watches step-ca's certificate transparency log (or receives a webhook/event from step-ca). Sees new cert for myapp.node-a.mesh.catalyst.local. Adds route to local RIB: FQDN → local upstream (webapp's cluster IP + port).
B7 orchestrator Writes xDS — SNI filter chain for service
Orchestrator updates /xds/lds.json: adds a new filter_chain on the :443 listener with server_names: ["myapp.node-a.mesh.catalyst.local"]. Updates /xds/sds.json: adds TLS context with webapp's cert + key (or a reference). Updates /xds/cds.json: adds upstream cluster pointing to webapp's local address. Envoy hot-reloads — service is now routable locally.
B8 orchestrator Writes DNS zone record
Orchestrator updates /zones/mesh.catalyst.local.db: adds myapp.node-a IN A <envoy-cluster-ip>. CoreDNS picks up the change via inotify — service is now DNS-resolvable locally.
Phase 3 — Mesh Propagation
B9 orchestrator → peers BGP UPDATE to all peers
Orchestrator sends BGP UPDATE message to all established peer sessions. Payload includes: FQDN (myapp.node-a.mesh.catalyst.local), nodePath (routing path), CA thumbprint (trust anchor), Cedar policy hints (allowed principals). Peers ACK.
B10 peer orchestrators Remote nodes update DNS + Envoy
Each peer's orchestrator receives the UPDATE, adds route to its RIB, writes zone file (FQDN → local Envoy, which will proxy via QUIC to originating node), writes xDS (CDS cluster for the QUIC upstream to node-a, LDS filter chain for SNI match). Service is now reachable mesh-wide.
B11 cedar-engine Cedar policies updated
Orchestrator pushes service-specific Cedar policies to the policy engine (both local and via BGP to peers). Default policy: deny all except principals with valid cert-bound tokens and matching FQDN scope. Envoy's ext_authz will evaluate these on every request.
Phase 1 — Data Channel Registration
B1 webapp / CLI Register data channel via orchestrator API
User or webapp calls the orchestrator's capnweb API: registerService({ fqdn: "myapp.node-a.mesh.catalyst.local", channel: { type: "http", upstream: "http://myapp:8080" } }). Route is added to the local RIB. This is the trigger — everything else follows.
B2 orchestrator Provisions DNS + xDS + TLS cert
On route_added event, orchestrator atomically: (1) writes DNS zone record (myapp.node-a IN A <envoy-ip>) — CoreDNS hot-reloads. (2) writes xDS (LDS filter chain with SNI match, CDS upstream cluster, SDS with node-managed cert). Envoy hot-reloads. (3) requests TLS cert from step-ca for the FQDN via IStepCaClient.issueCert(). Cert stored in CertCache. Service is now DNS-resolvable and routable locally.
Phase 2 — Service ACME Certificate (optional)
B3 webapp Webapp requests own ACME cert (optional)
Webapp uses a standard ACME client to contact step-ca's ACME directory. Requests a cert for myapp.node-a.mesh.catalyst.local. The FQDN already resolves (DNS was provisioned in B2), so the HTTP-01 challenge can complete. This step is optional — the node-managed cert from B2 is sufficient for mesh routing.
B4 step-ca → webapp step-ca validates HTTP-01 challenge
step-ca makes a GET request to the webapp at /.well-known/acme-challenge/<token>. DNS resolves because the orchestrator already provisioned the record. Challenge passes. step-ca signs the certificate. Webapp downloads cert + chain via ACME finalize.
B5 orchestrator Orchestrator updates SDS with service cert
If the webapp obtained its own cert, orchestrator updates /xds/sds.json to replace the node-managed cert with the service-managed cert. Envoy hot-reloads. Service now terminates TLS with its own identity.
Phase 3 — Mesh Propagation
B6 orchestrator → peers BGP UPDATE to all peers
Orchestrator sends BGP UPDATE message to all established peer sessions. Payload includes: FQDN (myapp.node-a.mesh.catalyst.local), nodePath (routing path), CA thumbprint (trust anchor), Cedar policy hints (allowed principals). Peers ACK.
B7 peer orchestrators Remote nodes update DNS + Envoy
Each peer's orchestrator receives the UPDATE, adds route to its RIB, writes zone file (FQDN → local Envoy, which will proxy via QUIC to originating node), writes xDS (CDS cluster for the QUIC upstream to node-a, LDS filter chain for SNI match). Service is now reachable mesh-wide.
B8 cedar-engine Cedar policies updated
Orchestrator pushes service-specific Cedar policies to the policy engine (both local and via BGP to peers). Default policy: deny all except principals with valid cert-bound tokens and matching FQDN scope. Envoy's ext_authz will evaluate these on every request.

Flow C First Client Connection — CLI Cert & mTLS to Service

A user on a remote node uses the Catalyst CLI to create a client certificate, obtains a certificate-bound JWT, then connects to the webapp via DNS resolution and mTLS through the Envoy mesh. Cedar policy evaluated at every hop.
Phase 1 — Client Identity
C1 CLI User generates key pair
User runs catalyst cert create --name alice. CLI generates an ECDSA P-256 key pair locally. Private key stays on disk (e.g. ~/.catalyst/alice.key). Creates a CSR with subject CN=alice.
C2 CLI → step-ca Certificate signing request via ACME
CLI uses ACME (or step-ca's direct API) to submit the CSR. step-ca signs a short-lived client cert: CN=alice, O=catalyst-mesh. Cert includes the public key thumbprint that will bind to tokens. CLI stores cert at ~/.catalyst/alice.crt.
C3 CLI → token-service Request certificate-bound JWT
CLI requests a JWT from the token service, presenting the client cert via mTLS. Token service verifies cert chain (signed by step-ca). Mints a JWT with cnf.x5t#S256 claim set to the SHA-256 thumbprint of alice's cert (RFC 8705). JWT includes scopes: ["myapp.node-a.mesh.catalyst.local"].
C4 CLI Client identity ready
Alice now has: (1) private key, (2) client cert signed by mesh CA, (3) JWT bound to that cert. These three together prove identity to any Envoy in the mesh.
Phase 2 — DNS Resolution
C5 client app → CoreDNS Resolve service FQDN
Client (curl, browser, any app) resolves myapp.node-a.mesh.catalyst.local. DNS query goes to local CoreDNS (either configured directly or via DHCP on appliance). CoreDNS returns the local Envoy's IP (the mesh routes everything through Envoy).
Phase 3 — Local Envoy — TLS + Cedar Policy
C6 client app → local Envoy mTLS handshake with local Envoy
Client connects to Envoy :443 with SNI = myapp.node-a.mesh.catalyst.local. Presents alice's client cert. Envoy matches SNI to filter chain. TLS handshake completes — Envoy has the client's cert identity + the target FQDN.
C7 local Envoy → Cedar Cedar policy evaluation (local hop)
Envoy calls ext_authz with: principal=CN:alice, resource=myapp.node-a.mesh.catalyst.local, action=CONNECT, token=<JWT>. Cedar verifies: JWT signature valid, cnf.x5t#S256 matches presented cert, principal is allowed to access resource. ALLOW — connection proceeds.
Phase 4 — QUIC Transit to Remote Node
C8 local Envoy → remote Envoy QUIC stream to peer cluster
Local Envoy looks up the CDS cluster for node-a. Opens (or reuses) a QUIC connection to remote Envoy on :8443. Traffic flows as a QUIC stream within the peer cluster. mTLS between Envoys using node identity certs. SNI + identity headers forwarded in the stream.
Phase 5 — Remote Envoy — TLS + Cedar Policy
C9 remote Envoy → Cedar Cedar policy evaluation (remote hop)
Remote Envoy terminates TLS (independently — not trusting the local Envoy's decision). Extracts client identity from forwarded headers/nested token. Calls Cedar ext_authz again with same principal + resource. Cedar evaluates remote-side policy. ALLOW — request forwarded to webapp.
C10 remote Envoy → webapp Request delivered to webapp
Remote Envoy forwards the request to the webapp's local cluster address. Webapp receives a plain HTTP request (Envoy terminated TLS). Identity available via x-forwarded-client-cert header if needed. End-to-end: DNS lookup → local Envoy (TLS + Cedar) → QUIC cluster → remote Envoy (TLS + Cedar) → webapp.
Dependency Chain Summary
System Ready requires
step-ca (root of trust) → token-service → Cedar → CoreDNS + Envoy (watchers) → orchestrator (writes config) → DHCP/NAT (appliance)
Service Onboarding requires
System ready + ACME client in webapp → step-ca issues cert → orchestrator detects → writes xDS + DNS → BGP propagates to peers System ready + data channel registration via orchestrator API → orchestrator writes DNS + xDS + issues cert → (optional) webapp ACME using live DNS → BGP propagates to peers
Client Connection requires
Service onboarded + CLI cert from step-ca + JWT from token-service → DNS resolves → Envoy mTLS + Cedar at every hop → QUIC to remote → delivered

What's Changing

Catalyst is moving from a service you code against to an environment you route through. Today, applications integrate via an SDK — they open a WebSocket, call capnweb RPCs, and register data channels on allocated ports. That works, but it couples every consumer to our protocol and makes Catalyst a dependency in your code.

The new model flips this. We use PKI + QUIC to build a peer-to-peer Envoy mesh, then layer SNI and domain-based routing on top — no port allocation needed. Since we already propagate FQDNs across the mesh, we now populate a DNS service (with proxy capabilities for internet-facing services) that any application can resolve against. Catalyst becomes a web proxy or NAT gateway: configure it as your default route, and traffic flows to peered services transparently.

Architecturally, we split from 2 planes into 3. The management plane (orchestrator, PKI, token, Cedar) stays private and handles policy. The data plane (Envoy, DNS, PIK, DHCP/NAT) carries traffic and is the public core. Token validation is Envoy-native via Cedar ext_authz; JWKS is served directly by the orchestrator. ACME is handled by step-ca in the management plane — no separate data-plane services needed. The new application plane (Zenoh, GraphQL, MediaMTX) holds opt-in services that ride the data plane but are no longer part of the core — they're just applications like any other.

Deployment simplifies too. K8s manages either an appliance (hardware or virtual, dropped onto a network like a proxy) or a sidecar (embedded in an existing cluster, hooking into CoreDNS and the K8s networking stack). Both modes use the same images and config — the difference is where you point the default route.
